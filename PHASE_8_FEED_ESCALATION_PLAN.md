*Authored: 2026-06-01. Place this file in the project root alongside CLAUDE.md and CLAUDE_HANDOVER.md.*

## 1. WHY THIS FEATURE EXISTS

The app currently has per-task activity feeds (TaskActivity) but no mechanism for important observations to travel upward or sideways across Work Packages, Divisions, or the whole organisation. This creates three operational problems:

1. **Knowledge silos** — conversations in tasks stay buried; lessons learned never propagate
2. **Untracked escalations** — important observations are raised verbally or in small chats but never followed up methodically
3. **Cross-division blindness** — a finding or safety observation in one division never reaches others who could benefit

This feature introduces a unified feed architecture with a formal escalation mechanism that routes flagged comments upward through a defined hierarchy, with Managers and Directors taking structured action on them.

## 2. COMPLETE DESIGN DECISIONS (LOCKED)

All decisions below were made during planning. Do not revisit them without a deliberate discussion — they are not open questions.

### 2.1 Feed Hierarchy

There are four feed scopes, forming a hierarchy:

```
Task Feed      (scope: TASK, scopeId: taskId)
    ↓ user escalates — picks target level
WP Feed        (scope: WP, scopeId: wpId)
    ↓ Manager disseminates
Org Feed       (scope: ORG, scopeId: null)

Division Board (scope: DIVISION, scopeId: divisionId)
    ↓ Manager disseminates
Org Feed       (scope: ORG, scopeId: null)
```

Division Board is a parallel track — not strictly above WP Feed. A task comment can be escalated to WP, DIVISION, or ORG directly. **Skipped levels always receive a non-actionable INFO_CARD** so nothing goes dark.

### 2.2 Post Types

| Type | Meaning |
|---|---|
| `COMMENT` | Human-authored message |
| `SYSTEM_EVENT` | Auto-generated by backend on status changes etc. |
| `ESCALATION_CARD` | A flagged comment surfaced at the target scope — actionable by Manager/Director |
| `INFO_CARD` | A read-only notice at a skipped scope — display only, no actions |

### 2.3 Read / Write / Action Rights

| Scope | Who can read | Who can post original | Who can action escalation flags |
|---|---|---|---|
| Task Feed | All authenticated users | All authenticated users | Manager (same div) + Director |
| WP Feed | All authenticated users | All authenticated users | Manager (same div) + Director |
| Division Board | All authenticated users | Any division member | Manager (same div) + Director |
| Org Feed | All authenticated users | Director + Managers only | Director + Managers |

### 2.4 Escalation Flow

1. Any authenticated user can flag any `COMMENT` on any feed level
2. User picks the target scope: `WP`, `DIVISION`, or `ORG`
3. An `EscalationFlag` record is created (status: `PENDING`)
4. An `ESCALATION_CARD` FeedPost is created at the target scope
5. An `INFO_CARD` FeedPost is created at every skipped scope between origin and target
6. Manager/Director reviews the flag and takes one action (see Section 2.5)
7. Flag moves to `ACTIONED` or `DISMISSED`

**Re-escalation upward** uses the existing Disseminate action — no second flag is created.

### 2.5 Flag Actions

| Action | Reuses existing workflow? | Notes |
|---|---|---|
| `ACKNOWLEDGED` | No (trivial) | Mark reviewed, no further action |
| `FINDING_RAISED` | ✅ Finding workflow | Pre-fills description from flagged comment |
| `DISSEMINATED` | Partial | Push card to Org Feed with optional division tags |
| `TASK_CREATED` | ✅ Task creation workflow | Pre-fills description from flagged comment |
| `REASSIGNED` | ✅ Task reassign workflow | Only valid if escalation originated from a task |

### 2.6 Disseminate Behaviour

When a Manager/Director disseminates to Org Feed:
- An `ESCALATION_CARD` is created at scope `ORG`
- They can optionally tag relevant divisions (`taggedDivisionIds`)
- The Org Feed defaults to showing all posts; users can filter to tagged-only
- Director always sees everything

### 2.7 Notifications

**Deferred to Phase 9.** In Phase 8, visibility is provided by **badge counts** on sidebar nav items only — same pattern as the existing Findings amber badge. No push, email, or WebSocket notifications in this phase.

### 2.8 Threading

**Flat feeds only** in Phase 8. No threaded replies. May be revisited in a future phase.

### 2.9 Info Cards

**Display-only.** No interaction, no acknowledgement button. They exist purely so intermediate levels are not oblivious to escalations that passed through.

### 2.10 TaskActivity Migration

`TaskActivity` is **replaced entirely** by `FeedPost`. Pre-production, so no data migration of production records is needed. All existing writes to `TaskActivity` are repointed to `FeedPost` with `scope: 'TASK'` and `scopeId` in place of `taskId`. Behaviour is identical from the outside.

## 3. DATA MODELS

### 3.1 FeedPost (replaces TaskActivity)

```prisma
model FeedPost {
  id                Int       @id @default(autoincrement())
  type              String    // COMMENT | SYSTEM_EVENT | ESCALATION_CARD | INFO_CARD
  scope             String    // TASK | WP | DIVISION | ORG
  scopeId           Int?      // taskId | wpId | divisionId | null for ORG
  authorId          Int?      // null for SYSTEM_EVENT entries
  content           String
  sourcePostId      Int?      // ESCALATION_CARD + INFO_CARD point to origin FeedPost
  sourceExcerpt     String?   // denormalized excerpt — no join needed for display
  sourceTaskId      Int?      // denormalized origin task ID for breadcrumb/link
  sourceWpId        Int?      // denormalized origin WP ID for breadcrumb/link
  flagId            Int?      // link to EscalationFlag that spawned this card
  taggedDivisionIds Json?     // [divisionId, ...] — Org Feed tagging only
  metadata          Json?     // { fromStatus, toStatus, extensionDecision, etc. }
  createdAt         DateTime  @default(now())

  author            User?     @relation(fields: [authorId], references: [id])

  @@index([scope, scopeId])
}
```

**Field mapping from old TaskActivity:**

| Old (TaskActivity) | New (FeedPost) | Notes |
|---|---|---|
| `taskId` | `scopeId` | Always paired with `scope: 'TASK'` for existing writes |
| `authorId` | `authorId` | Unchanged |
| `type` | `type` | Existing values COMMENT, SYSTEM_EVENT still valid |
| `content` | `content` | Unchanged |
| `metadata` | `metadata` | Unchanged |
| *(none)* | `scope` | Always `'TASK'` for all Phase 8.0 writes |

### 3.2 EscalationFlag

```prisma
model EscalationFlag {
  id                Int       @id @default(autoincrement())
  sourcePostId      Int
  flaggedByUserId   Int
  reason            String?
  targetScope       String    // WP | DIVISION | ORG
  status            String    @default("PENDING") // PENDING | ACTIONED | DISMISSED
  reviewedByUserId  Int?
  action            String?   // ACKNOWLEDGED | FINDING_RAISED | DISSEMINATED | TASK_CREATED | REASSIGNED
  actionedAt        DateTime?
  linkedEntityId    String?   // Finding ID or Task ID created from this flag
  createdAt         DateTime  @default(now())

  flaggedByUser     User      @relation("FlaggedBy", fields: [flaggedByUserId], references: [id])
  reviewedByUser    User?     @relation("ReviewedBy", fields: [reviewedByUserId], references: [id])
}
```

### 3.3 User model relations to add

The `User` model in schema.prisma will need these back-references added:

```prisma

flagsRaised       EscalationFlag[] @relation("FlaggedBy")
```

```typescript

  // For DISSEMINATED:

  findingOverride?: { eventType?: string, departmentId?: number, description?: string },
  // For TASK_CREATED:
  taskOverride?: { templateId: number, title?: string, assignedToUserId?: number },
  // For REASSIGNED:
  newAssigneeId?: number,
  reason?: string
}
```

## 6. RBAC RULES FOR FEED ENDPOINTS

### Read access
All authenticated users can read all feed scopes. Consistent with existing system-wide transparency model.

### Write access (original posts)
- `TASK` scope: any authenticated user
- `WP` scope: any authenticated user
- `DIVISION` scope: any member of that division
- `ORG` scope: Director or Manager only

### Escalation (flagging)
Any authenticated user can flag any `COMMENT` post.

### Flag actions
- `WP` scope flags: Manager of same division + Director
- `DIVISION` scope flags: Manager of same division + Director
- `ORG` scope flags: Director + any Manager (they may not be from the originating division)

### Division membership check
Use existing `divisionId` on the User model. A user is a member of a division if `user.divisionId === division.id`. Directors bypass all division checks.

## 7. DUAL WRITE RULE — UNCHANGED

The existing rule applies to all FeedPost writes exactly as it did for TaskActivity:

> Every status change or significant event writes to BOTH `AuditLog` (compliance) AND `FeedPost` with `type: 'SYSTEM_EVENT'`. Never one without the other.

Escalation events (flag created, flag actioned) also write to `AuditLog`.

```
function escalatePost(postId, targetScope, flaggedByUserId, reason):

  3. Create EscalationFlag (status: PENDING)
```

- Source: DIVISION, Target: ORG → no skipped scopes

## 9. KNOWN GOTCHAS & DECISIONS TO CARRY FORWARD

1. **User model relations** — Adding `EscalationFlag` introduces two named relations on `User` (`FlaggedBy`, `ReviewedBy`). Prisma will error if the back-references are not added to the `User` model simultaneously. Always add User back-references in the same schema edit as the new model.

2. **Old activity endpoints** — `GET /api/tasks/:id/activity` and `POST /api/tasks/:id/activity` are used by the existing frontend. Do not remove them until Phase 8.3 updates the frontend. In Phase 8.1, they can be left as-is (they already use FeedPost after the 8.0 migration) or proxied to the new feed endpoints.

3. **scopeId is nullable** — Org Feed posts have `scopeId: null`. All queries against Org Feed must use `where: { scope: 'ORG' }` without a scopeId filter.

4. **INFO_CARD creation requires wpId** — When escalating from a TASK post to DIVISION or ORG, creating an INFO_CARD on the WP Feed requires knowing the task's `wpId`. If the task has no WP, skip the WP-level INFO_CARD silently.

5. **Dual write still applies** — Every escalation event (flag created, flag actioned) writes to AuditLog. The FeedPost creation is the operational record; AuditLog is the compliance record.

6. **taggedDivisionIds filter default** — Org Feed always defaults to showing ALL posts. The division tag filter is opt-in. Never change this default.

7. **EscalationFlag is not soft-deleted** — Flags are compliance records. Once created, they are never physically or soft deleted. Status moves to ACTIONED or DISMISSED only.

8. **Finding pre-fill on FINDING_RAISED action** — The flag action handler calls the existing finding creation logic. It should pre-fill `description` from `sourceExcerpt` and `sourceTaskId` from the flag's source post. The reviewer can override these in the action request body. Do not bypass the existing Finding validation (eventType and departmentId are still required).

9. **TASK_CREATED action requires a templateId** — Creating a task always requires a published template. The flag action request for TASK_CREATED must include `templateId`. The frontend FlagActionPanel must show a template picker for this action.

10. **Test baseline** — As of Phase 8.0 completion: 187 tests. Each sub-phase adds tests. Never reduce the count.

11. **EscalationFlag has no `sourcePost` relation in Prisma** — Despite `sourcePostId` existing as a field, there is no declared Prisma relation. Load the source FeedPost separately via `prisma.feedPost.findUnique({ where: { id: flag.sourcePostId } })`. Do not attempt `include: { sourcePost: true }` on EscalationFlag.

12. **Template has no `deletedAt` column** — The soft-delete pattern does not apply to Template. Filter Published templates using `status: 'Published'` only. Never add `deletedAt: null` to Template queries.
